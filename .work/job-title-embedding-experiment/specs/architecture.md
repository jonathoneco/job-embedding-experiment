# Architecture — Job Title Embedding Matching Experiment

## Problem Statement

The recommendation system needs to match blurry, user-defined job titles against a maintained taxonomy of 692 roles across 42 categories. This experiment verifies whether direct embedding comparison is a valid approach, and determines the optimal combination of embedding model tier and comparison granularity.

## Goals

1. Generate 750 realistic test cases covering abbreviations, synonyms, creative titles, jargon, and misspellings
2. Compare 3 embedding models across 5 granularity levels (15 configurations)
3. Include 3 non-embedding baselines (TF-IDF, fuzzy, BM25) for validity context
4. Provide statistical rigor (bootstrap CIs, McNemar's test)
5. Produce a reviewable report with clear recommendations
6. Run embedding compute on garden-pop (4080 GPU) via SSH

## Open Questions Resolved

### 1. Python project structure
**Decision**: Flat modular structure under `src/` with a single orchestrator script. Not a pip-installable package — this is a one-shot experiment, not a library. Dependencies managed via `pyproject.toml` with `uv` for reproducible installs.

### 2. Role descriptions for Role+Desc and Category+Desc levels
**Decision**: LLM-generated via Claude API. One-time batch generation, output committed to `data/taxonomy/descriptions.json`. Each role gets a 10-15 word functional description (e.g., "HR Business Partner" → "Strategic HR advisor who aligns people strategy with business objectives"). Category descriptions aggregate key terms from their roles. Generated once, reviewed via spot-check, then frozen.

### 3. Result format
**Decision**: Markdown report (`results/report.md`) with embedded PNG charts generated by matplotlib/seaborn. Reasons: easy to review in any tool, versionable in git, no notebook rendering issues. Charts saved to `results/figures/` and referenced via relative paths.

### 4. Reproducibility
**Decision**:
- `pyproject.toml` with pinned dependency versions
- Config file (`config.yaml`) with random seeds, model revision hashes, batch sizes
- Data files committed to git (except embedding cache, which is gitignored)
- Single `run_experiment.py` orchestrator with deterministic execution order

## Component Map

### C0: Project Setup
**Scope**: Directory structure, dependencies, configuration, gitignore
**Effort**: Small
**Dependencies**: None

- `pyproject.toml` — dependencies: sentence-transformers, scikit-learn, rapidfuzz, rank-bm25, matplotlib, seaborn, numpy, anthropic, pyyaml, pytest
- `config.yaml` — seeds, model IDs with HuggingFace revision hashes, batch sizes, file paths
- `.gitignore` — cache directories, embedding files, `__pycache__`

### C1: Taxonomy & Targets
**Scope**: Parse taxonomy, build clusters, generate descriptions, construct target sets
**Effort**: Medium
**Dependencies**: C0

Subcomponents:
- **Taxonomy parser** (`src/taxonomy.py`): Parse `job-roles.md` into structured data — list of `{role, category}` dicts. Output: `data/taxonomy/roles.json`
- **Cluster builder** (`src/clusters.py`): Category-scaffold approach — split categories with 10+ roles into 2-4 subclusters with manual labels. Output: `data/taxonomy/clusters.json`
- **Description generator** (`src/descriptions.py`): Claude API batch call to generate one-line descriptions for all 692 roles and 42 categories. Output: `data/taxonomy/descriptions.json`
- **Target builder** (`src/targets.py`): Construct the 5 target sets from the above data:
  1. Role: `[role_name, ...]` (692 targets)
  2. Role+Desc: `[f"{role_name}: {description}", ...]` (692 targets)
  3. Cluster: `[cluster_label, ...]` (80-120 targets)
  4. Category+Desc: `[f"{cat_name}: {key_terms}", ...]` (42 targets)
  5. Category: `[cat_name, ...]` (42 targets)

  Each target set is a list of `{id, text, role, category}` dicts where `text` is what gets embedded.

### C2: Test Data
**Scope**: Generate test cases, validate, split into dev/test
**Effort**: Large (most manual effort)
**Dependencies**: C1 (needs taxonomy for validation)

Difficulty distribution targets: **250 easy, 325 medium, 175 hard** (750 total).

| Method | Easy | Medium | Hard | Total |
|--------|------|--------|------|-------|
| Rule-based transforms | 120 | — | — | 120 |
| LLM systematic (Pass 1) | 105 | 200 | — | 305 |
| LLM adversarial (Pass 2-3) | — | 125 | 125 | 250 |
| Manual curation | — | — | 50 | 50 |
| O*NET synonyms | 25 | — | — | 25 |

Overflow strategy: if LLM generates more than needed for a tier, trim by removing the most similar cases (deduplicate by cosine similarity on input titles).

Subcomponents:
- **Rule-based generator** (`src/generate_rules.py`): Abbreviation expansion, level prefix/suffix transforms, minor rewordings. 120 easy cases.
- **LLM generator** (`src/generate_llm.py`): Multi-pass Claude API generation — systematic pass (305 cases), adversarial passes (250 cases). Structured output with variation type tagging.
- **Manual cases** (`data/test-cases/manual.json`): 50 hard cases curated by hand. Plus 25 O*NET synonym cases.
- **Validator** (`src/validate.py`): Schema validation, deduplication (cosine similarity on inputs), taxonomy membership check, difficulty distribution verification.
- **Splitter**: Random stratified split into dev (100) and test (650), preserving difficulty and category distribution.

Output: `data/test-cases/dev.json`, `data/test-cases/test.json`

Schema per case:
```json
{
  "id": "TC-0001",
  "input_title": "Sr. HRBP",
  "correct_roles": [
    {"role": "HR Business Partner", "category": "Human Resources"}
  ],
  "difficulty": "easy",
  "variation_type": "abbreviation",
  "source": "rule-based",
  "notes": "Standard abbreviation"
}
```

### C3: Matching Pipeline
**Scope**: Embedding models, baseline methods, similarity computation
**Effort**: Medium
**Dependencies**: C0, C1 (target sets)

Subcomponents:
- **Embedding engine** (`src/embed.py`): Load model via `sentence-transformers`, encode target sets and query inputs, compute cosine similarity matrix. Returns ranked results per query. Supports caching embeddings to disk (`data/embeddings/<model>/`).
- **Baseline methods** (`src/baselines.py`):
  - TF-IDF: Character n-gram (3-5) vectorizer from scikit-learn, cosine similarity
  - Fuzzy: Jaro-Winkler via rapidfuzz, returns similarity scores
  - BM25: rank-bm25 library, tokenized on words
  Each baseline produces the same output format as the embedding engine: ranked results per query.
  Baselines run against the **Role** and **Category** target sets only (2 granularity levels). Description-enriched and cluster targets don't meaningfully help text-matching baselines, so testing all 5 would add noise without insight.

Models (from config):
- `all-MiniLM-L6-v2` — 384-dim
- `BAAI/bge-base-en-v1.5` — 768-dim
- `BAAI/bge-large-en-v1.5` — 1024-dim

### C4: Evaluation & Reporting
**Scope**: Metrics, statistics, visualization, final report
**Effort**: Medium
**Dependencies**: C2, C3

Subcomponents:
- **Metrics engine** (`src/evaluate.py`): Compute per-configuration:
  - MRR (Mean Reciprocal Rank)
  - Top-1, Top-3, Top-5 accuracy
  - Category accuracy (correct category regardless of role)
  - Mean similarity gap (score of rank-1 minus rank-2)
  All metrics respect accept-set ground truth (any match in `correct_roles` counts).
- **Statistical analysis** (`src/statistics.py`):
  - Bootstrap CIs (1000 resamples) for all metrics
  - McNemar's test for pairwise model comparison
  - Friedman + Nemenyi for multi-model ranking (if needed)
- **Report generator** (`src/report.py`): Generate markdown report with:
  - Executive summary with recommendation
  - Heatmap: model x granularity → MRR
  - Bar charts per metric grouped by model
  - Difficulty breakdown (easy/medium/hard)
  - Statistical significance tables
  - Error analysis: 10 worst failures per best model
  - Production threshold check (MRR >= 0.75, Top-3 >= 0.85)

Output: `results/report.md`, `results/figures/*.png`, `results/metrics/*.json`

## Data Flow

```
job-roles.md
     │
     ▼
[C1] taxonomy.py ──→ roles.json (692 roles)
     │
     ├──→ clusters.py ──→ clusters.json (80-120 subclusters)
     ├──→ descriptions.py ──→ descriptions.json (692 descriptions)
     └──→ targets.py ──→ target_sets.json (5 granularity levels)

[C2] generate_rules.py ─┐
     generate_llm.py ────┼──→ raw_cases.json ──→ validate.py ──→ dev.json + test.json
     manual.json ────────┘

[C3] embed.py ──────────┐
     baselines.py ──────┼──→ rankings.json (per model × granularity)
                        │    {query_id, method, granularity, ranked_results}

[C4] evaluate.py ──→ metrics.json ──→ statistics.py ──→ significance.json
     report.py ──→ report.md + figures/*.png
```

## Execution Model

### Prep Phase (local machine, no GPU)
1. Parse taxonomy → `roles.json`
2. Build clusters → `clusters.json` (requires manual subclusters definition)
3. Generate descriptions → `descriptions.json` (Claude API)
4. Build target sets → `target_sets.json`
5. Generate test data → `raw_cases.json` → validate → `dev.json` + `test.json`

### Compute Phase (garden-pop, 4080 GPU)
6. Copy project to garden-pop (rsync or git clone)
7. Run `python scripts/run_experiment.py` — this:
   a. Loads targets and test cases
   b. For each model: encode + rank (cached to `data/embeddings/`)
   c. For each baseline: compute rankings
   d. Computes all metrics
   e. Runs statistical tests
   f. Generates report

Expected runtime: <5 minutes total (692 targets × 750 queries × 3 models = trivial on GPU).

**Error handling**: Fail-closed — the orchestrator halts on any error (Python's default exception propagation). No silent fallbacks, no partial results. If model loading, API calls, or validation fail, the script exits with a traceback.

### Review Phase (any machine)
8. Copy results back (rsync)
9. Review `results/report.md` and share with team

## Technology Choices

| Choice | Rationale |
|--------|-----------|
| Python 3.11+ | Standard for ML, sentence-transformers requires it |
| `uv` for dependency management | Fast, reproducible, lockfile support |
| `sentence-transformers` | Unified API for all 3 HuggingFace models |
| `scikit-learn` | TF-IDF vectorizer, metrics utilities |
| `rapidfuzz` | Fast Jaro-Winkler implementation |
| `rank-bm25` | Simple BM25 implementation, no server needed |
| `matplotlib` + `seaborn` | Static charts for markdown report |
| `anthropic` | Claude API for description + test data generation |
| `pyyaml` | Config file parsing |
| `pytest` | Unit tests for taxonomy parsing, metrics computation |
| Markdown report (not notebook) | Reviewable anywhere, versionable, no render issues |

## Scope Exclusions

- **No web UI or dashboard** — static markdown report only
- **No model fine-tuning** — using off-the-shelf models
- **No real-time serving** — batch evaluation only
- **No database** — JSON files for all data
- **No CI/CD** — manual execution on garden-pop
- **No Docker** — direct Python environment on garden-pop
- **No embedding-based clustering** — category scaffold approach avoids bootstrap problem
